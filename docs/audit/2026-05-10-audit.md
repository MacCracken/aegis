## F-1 â€” Unbounded `max_events` from JSON deserialization

**Severity**: MEDIUM (DoS) | **Status**: Fixed in 0.9.3.

**Root cause**: `aegis_config_from_json_v` reads `max_events` straight from the wire JSON int with no upper bound. The value flows to `aegis_new(config) â†’ aegis_ring_new(config_max_events(config)) â†’ alloc(cap * 8)`. A consumer accepting `AegisConfig` JSON from an untrusted source can request a heap allocation of arbitrary size (e.g. `max_events = 10^15` â†’ ~8 PB request â†’ process kill).

**Exposure**: Direct hit. No upstream consumer hardens this today (consumers in plan: daimon, argonaut).

**Fix (0.9.3)**: Introduce `AEGIS_MAX_EVENTS_HARD_CAP = 1_000_000` (8 MB upper bound on the ring slot array). Clamp at deserialization in `aegis_config_from_json_v`. Same clamp applied in `aegis_config_set_max_events` (the setter â€” defense-in-depth).

**Test**: `test_aegis_config_max_events_clamped` â€” feed JSON with a huge `max_events`, assert deserialized value â‰¤ hard cap.

**CVE family**: CWE-770 Allocation of Resources Without Limits.

## F-2 â€” nftables identifier injection via `agent_id` in firewall builders

**Severity**: HIGH | **Status**: Fixed in 0.9.3.

**Root cause**: `aegis_isolate_agent(agent_id, agent_addr)` and `aegis_rate_limit_agent(...)` interpolate `agent_id` into table names (`aegis_iso_<agent_id>`, `aegis_rl_<agent_id>`) and into rule comments (`aegis isolate <agent_id>`). nein's `validate_identifier` rejects non-`[a-zA-Z0-9_-]` only at `firewall_validate` time â€” *after* the firewall is built. Worse: rule comments are not subjected to `validate_identifier` at all in nein 1.5.0; an `agent_id` containing `\n`, `"`, `;`, `{`, or `#` could escape the comment context in the rendered nftables source. The iptables-save newline write-primitive (Shielder, 2024) is the canonical example of this class â€” there a sudo'd user forged `/etc/passwd` lines via an iptables comment.

**Exposure**: Direct hit. Aegis's API-boundary contract today is "cstrs at the seam, no validation"; the only check is downstream in nein, which the caller may not even invoke (`aegis_firewall_render` does not call `aegis_firewall_validate` first).

**Fix (0.9.3)**: Add `_aegis_valid_agent_id(cstr) â†’ 0/1` (whitelist `^[a-zA-Z0-9_.-]{1,54}$` â€” 54 chars to keep `aegis_iso_<id>` â‰¤ 64-char nein limit). `aegis_isolate_agent` / `aegis_rate_limit_agent` return `0` (null fw pointer) on invalid input.

**Note on quarantine path**: The same `agent_id` flows into the quarantine map (`aegis_quarantine_agent`, `aegis_is_quarantined`, `aegis_get_quarantine`, `aegis_release_agent`). Adding validation on those paths is an API-contract change â€” see F-7. For 0.9.3 we tighten only the firewall seam where injection is concretely demonstrable.

**Test**: `test_firewall_isolate_rejects_bad_agent_id` â€” call with `"agent\n; drop"`, assert returned fw is 0; same for empty + over-length.

**CVE family**: CWE-77 Command Injection (in the rule-text generation context); analogue of iptables-save comment write-primitive.

## F-3 â€” nftables match-value injection via `agent_addr`

**Severity**: HIGH | **Status**: Fixed in 0.9.3.

**Root cause**: Same builders accept `agent_addr` and pass it raw to `match_dest_addr(addr)` / `match_source_addr(addr)`. nein 1.5.0 does not validate the address format â€” the cstr is stored and rendered as-is. A bad `agent_addr` like `"10.0.0.1\n drop\n#"` could inject nftables tokens into the rendered ruleset.

**Exposure**: Direct hit, same as F-2.

**Fix (0.9.3)**: Add `_aegis_valid_agent_addr(cstr) â†’ 0/1` â€” pragmatic shape validator: characters limited to `[0-9a-fA-F:.]` plus optional `/[0-9]+` CIDR suffix; total length 1..43 (max IPv6 with `/128`). This is not a full RFC 4291 parse but rejects every nftables-injection vector. Builders return `0` on invalid input.

**Deeper fix deferred**: A typed parse (4-byte IPv4 / 16-byte IPv6) with canonical re-emit is the right long-term shape â€” moves to 0.9.5+.

**Test**: `test_firewall_isolate_rejects_bad_agent_addr` â€” call with `"10.0.0.1\n; drop"`, assert returned fw is 0; spot-check IPv4 + IPv6 + CIDR pass.

**CVE family**: CWE-77 (rule-text injection).

## F-4 â€” Sentinel collision in `auto_release_timeout_secs` from JSON

**Severity**: LOW | **Status**: Fixed in 0.9.3.

**Root cause**: `AEGIS_AUTO_RELEASE_NONE = -1` is the documented sentinel for "no auto-release". Other negative values from the wire (e.g. `-86400`) silently parse and flow to `clock_epoch_secs() + timeout` â†’ past epoch â†’ quarantine entry is auto-released on the very next `aegis_check_auto_releases` call, regardless of severity.

**Exposure**: Low because requires consumer trusting attacker-controlled `AegisConfig`. But it's a quarantine-bypass via type-system gap.

**Fix (0.9.3)**: In `aegis_config_from_json_v`, accept only `-1` (sentinel) or `[0, AEGIS_TIMEOUT_HARD_CAP]` (1 year of seconds = 31_536_000). Out-of-range â†’ use the default.

**Test**: `test_aegis_config_auto_release_negative_rejected` â€” JSON with `-86400`, assert deserialized to default.

**CVE family**: CWE-690 (Semipredicate / sentinel collision).

## F-6 â€” TOCTOU + symlink follow in scanner

**Severity**: HIGH | **Status**: **Fixed in 0.9.5**.

**0.9.4 update (kept for history)**: The "cheap fix" path (switch `sys_stat` â†’ `sys_lstat`) is blocked on cyrius stdlib â€” neither `lib/syscalls_x86_64_linux.cyr` nor `lib/syscalls_aarch64_linux.cyr` exposes `sys_lstat` or `AT_SYMLINK_NOFOLLOW`. Hardcoding raw `syscall(6, ...)` in aegis source while every other syscall flows through the stdlib's `SYS_*` constants is an inconsistency we don't want to lock in just before 1.0.0.

**0.9.5 fix**: Skipped the lstat path entirely; landed the openat-style refactor that uses syscalls already in the cyrius stdlib. `_aegis_stat_modesize(path, out)` now does `sys_open(path, O_RDONLY | O_CLOEXEC | O_NOFOLLOW, 0)` â†’ `sys_fstat(fd, sb)` â†’ `sys_close(fd)`. On a symlink the open returns `-ELOOP`; aegis returns `0` (failure). Existing scanner failure paths catch this and emit an `unreadable_metadata` finding (file_exists still uses a follow-symlink open, so symlinks pass that check; `_aegis_stat_modesize` then trips the F-6 gate, landing in the existing "metadata unreadable" branch).

**Local `_AEGIS_O_NOFOLLOW = 131072` constant**: stable POSIX flag value on every Linux arch aegis targets (x86_64 + aarch64 â€” verified against `include/uapi/asm-generic/fcntl.h`). The cyrius stdlib didn't expose it; defined locally with an upstream-this comment so future contributors know to remove the local constant once `O_NOFOLLOW` lands in `lib/syscalls_*_linux.cyr`. Single grep target.

**Behaviour change**: scanning a symlinked binary path now flags it as `unreadable_metadata` instead of dereferencing to the target. Marked `### Breaking` in the 0.9.5 CHANGELOG entry (second `### Breaking` in 0.9.x â€” pre-1.0 contract changes are acceptable per Keep-a-Changelog conventions; 1.0.0 freezes the surface).

**Test**: `test_p1_scan_refuses_to_follow_symlink` â€” sets up a real file + symlink â†’ real file via `sys_symlink`, scans both: real file scans clean, symlink scan surfaces `unreadable_metadata` finding. Cleanup at end.

**Deeper future fix (post-1.0 if needed)**: Pass the open fd to the consumer as part of the SecurityFinding bundle so the consumer's action operates on the same inode aegis stat'd. Requires API change to SecurityFinding â€” out of scope for the v1.0 cut.

**Root cause**: `_aegis_stat_modesize(path_cstr, out)` uses `sys_stat(path)` â€” follows symlinks, and the path is re-resolved on each call. The textbook anti-pattern (CERT FIO45-C / POS35-C). Recent CVE shape: ESET CA8840 / **CVE-2025-2425** (EDR TOCTOU), **CVE-2025-22224** (VMware ESXi TOCTOU OOB write), **CVE-2024-50379** (Tomcat TOCTOU). An agent that symlinks its `binary_path` to a clean file at scan-enroll, then swaps the symlink target to a malicious file before the consumer acts on the scan finding, defeats the integrity check.

**Aegis-specific mitigation**: Aegis only *stats* â€” it doesn't read file content â€” so the in-aegis TOCTOU window is narrow (between stat and store of mode/size). The exploitable window is between aegis returning a finding and the *consumer* acting on it. Aegis cannot fully close that, but it can:

1. **0.9.4 cheap fix**: switch `sys_stat` â†’ `sys_lstat`; if the result is `S_ISLNK`, surface a finding ("symlink in scan path â€” re-resolve before action") instead of silently following.
2. **0.9.5 deeper fix**: rework the scanner around `sys_open(path, O_RDONLY|O_NOFOLLOW|O_CLOEXEC)` followed by `sys_fstat(fd)`. Open the fd once; pass the fd (or a finding bundle including it) to the consumer.

**Why deferred**: The lstat swap requires test rework (the existing tests use plain files; symlink-handling tests need `sys_symlink` plumbing in the test harness). The deeper openat-rework requires syscall additions. Both fit cleanly into 0.9.4 as a focused slice.

**Tracker**: TaskList #44 covers the lstat swap; #45 covers the openat refactor.

**CVE family**: CWE-367 Time-of-check Time-of-use Race Condition.

## F-7 â€” Unicode-normalization quarantine bypass via `agent_id`

**Severity**: MEDIUM (HIGH if attacker controls registration) | **Status**: **Fixed in 0.9.4**.

**Root cause**: The cstr-keyed quarantine hashmap (`map_new()`) does byte-for-byte key comparison via `streq`. An attacker who can register agents (or supply IDs to a consumer that quarantines them) can bypass the lookup by using:

- **Case variants**: `agent-1` vs `Agent-1` â€” two distinct keys.
- **Unicode confusables**: `agentâ€‘1` (U+2011 NON-BREAKING HYPHEN) vs `agent-1` (U+002D HYPHEN-MINUS).
- **Zero-width insertions**: `agentâ€‹-1` vs `agent-1`.
- **Right-to-left override**: U+202E in the middle of an ID.

Recent CVE shape: **CVE-2024-43093** (Android, CISA KEV â€” `shouldHideDocument` filter bypassed via Unicode normalization mismatch), **CVE-2025-52488** (DNN, U+FF0E / U+FF3C bypass), **CVE-2024-38820/38827** (Spring locale-dependent `toLowerCase()` bypass).

**Fix (0.9.4)**: `_aegis_valid_agent_id` (introduced in 0.9.3 for the firewall path) moved from `src/firewall.cyr` to `src/lib.cyr` so both modules share the validator. Applied at: `aegis_quarantine_agent` / `aegis_release_agent` / `aegis_is_quarantined` / `aegis_get_quarantine` (all return 0 on invalid input â€” matches "key not in map" semantics) and `security_event_new` (variant `_aegis_valid_agent_id_or_empty` allows null/empty for anonymous events; non-empty bad ids strip to `aid=0` and the event is still recorded â€” silently dropping events would mask attack attempts).

**Test**: 5 new test groups in `tests/aegis.tcyr` (19 assertions) covering each entry point + the regression-guard for valid input + the security_event strip-bad-id behaviour.

**CVE family**: CWE-176 Improper Handling of Unicode Encoding; CWE-178 Improper Handling of Case Sensitivity.

## F-8 â€” JSON depth-exhaustion DoS

**Severity**: HIGH | **Status**: **Partial fix in 0.9.3** (input-length cap); deeper depth cap deferred.

**Root cause**: `lib/json.cyr`'s typed-value tree parser (`json_v_parse`) is recursive by construction. Deeply nested input (`{"a":{"a":{"a":...}}}`) exhausts the stack. Industry analogues: **CVE-2025-52999** (Jackson Core, CVSS 8.7), **CVE-2024-57699** (json-smart), **CVE-2024-21907** (Newtonsoft.Json), **CVE-2024-34155** (Go encoding/json) â€” all the same class, all fixed by hard depth caps (Jackson default 1000).

**Exposure**: Aegis's eight `*_from_json` entry points all call `json_v_parse` directly. Any consumer feeding wire JSON to aegis is exposed.

**Fix (0.9.3 â€” partial)**: At each aegis `*_from_json` seam, wrap with an input-length cap: `AEGIS_JSON_MAX_BYTES = 262_144` (256 KB). Inputs exceeding the cap return 0 (parse failure). This is a **partial mitigation** â€” depth-N nesting requires at least 2N bytes, so a 256 KB cap bounds maximum depth at ~131K nodes. That's not a tight depth cap, but it bounds the worst case to "process can fail loudly via overcommit/OOM rather than crash with a SIGSEGV from stack overflow at depth 100K+."

**Deeper fix (deferred)**: A real depth cap (suggest 32 â€” record nesting in aegis is shallow: `metadata` and `events` arrays at most one level deep) requires changing `lib/json.cyr` upstream. Filed at TaskList #45.

**Test**: `test_security_event_from_json_oversize_rejected` â€” feed a 300 KB JSON, assert returns 0.

**CVE family**: CWE-674 Uncontrolled Recursion.

## F-9 â€” CWE-690 sentinel collisions across record fields

**Severity**: MEDIUM | **Status**: **Fixed in 0.9.4** (annotation pass).

**Root cause**: Aegis follows ADR 0001 â€” per-field sentinels (`-1` / `0` / `QA_NONE`). The deliberate design is documented; the risk is that *future* fields added without going back to the ADR will silently collide. Concretely:

- `event_id` is a Str ptr; "absent" is 0 (null ptr). A heap allocation that happens to land at 0 (impossible in practice with mmap/brk allocators, but worth noting in ADR 0001 as a precondition).
- `threat_level` ranges 0..4; comparison `level < threshold` uses bare `<` (ADR 0001 rationale). If `threshold` is set to `-1` to mean "never quarantine", `level < -1` is always false â†’ quarantine never fires. Today `threshold` is hardcoded; if it ever becomes config-driven, validate.
- `auto_release_at` uses `-1` sentinel (covered by F-4).

**Fix (0.9.4)**: Preventive annotation pass.

- Introduced `_AEGIS_SERDE_INVALID = -1` named constant; replaced 4 magic-number `return 0 - 1` returns in `*_from_serde` parsers with the named form. Caller convention `if (parsed >= 0) { use(parsed); }` documented at the constant definition.
- `AEGIS_AUTO_RELEASE_NONE = -1` already had a docstring above its definition (added in 0.5.0); confirmed during this pass.
- `QA_NONE = 0` is documented as the "no action" sentinel in the `QuarantineAction` enum block.
- Re-affirmed precondition: aegis allocations never land at address 0 (relies on bump-allocator semantics from `lib/alloc.cyr`); a future allocator swap would need to preserve this.

No active code-behaviour change. The named-constant swap is a readability + grep-ability win that will catch any future enum-value-collides-with-sentinel addition at code-review time.

**CVE family**: CWE-690 Semipredicate / sentinel collision.

## What aegis got *right* (worth recording)

A P(-1) report should also say what's load-bearing and audit-clean today:

- **No `sys_system` calls** anywhere in src/ (audit gate enforces this).
- **No raw `execve` (syscall 59)** anywhere in src/.
- **No `/etc/shadow` access** anywhere in src/.
- **Cstrs at API boundary, Str* in storage** â€” ADR 0002 enforced. No silent type-confusion paths between the two.
- **Hashmap flavor** correctly chosen â€” `map_new()` (cstr-keyed) project-wide; never `map_new_str()` (ADR 0004).
- **Ring buffer** for events log is fixed-cap by construction; `aegis_ring_push` overwrites oldest at cap (ADR 0005); the F-1 risk is on the cap *value*, not the ring algorithm.
- **No format-string vulnerabilities**: all logging via `str_builder_add_cstr` / `_add_int` â€” explicit non-printf style.
- **Function-scope `var buf[N]` audit clean**: 5 sites (`var sb[144]` for stat, four `var ms[16]` for mode/size out-params); all bounded, no escapes.
- **Fuzz harness** (1000 random + curated inputs) covers all 8 record-from-json parsers; no crashes since 0.8.2.
- **Lint warnings: 0** across both source files.
